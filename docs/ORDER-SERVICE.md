# Бизнес-описание домена «Order Service»

Документ описывает бизнес-сущности, правила и сценарии Order Service в рамках [архитектуры проекта](DESIGN.md). Фокус — Saga (Choreography), жизненный цикл заказа и взаимодействие с Store и Payment.

---

## Бизнес-сущности

### 1. Корзина (Cart)

Временный набор товаров пользователя. Хранится в Redis. После успешного оформления заказа очищается.

**Атрибуты:**
- Идентификатор пользователя
- Список позиций (товар, количество, цена за единицу)
- Дата последнего обновления

**Правила:**
- Количество каждого товара — целое число ≥ 1.
- Цена за единицу > 0.
- При добавлении товара Order Service публикует событие `RESERVE_ITEMS` в NATS (Store резервирует остаток асинхронно).
- После подтверждения резерва (`ITEMS_RESERVED`) корзина считается готовой к оформлению.
- Оформление заказа возможно только из непустой корзины с подтверждённым резервом.

**Примечание:** при повторном добавлении того же товара — увеличивать количество (merge), а не создавать дубликат позиции.

---

### 2. Заказ (Order)

Центральная сущность. Персистентное хранение — PostgreSQL.

**Атрибуты:**
- Идентификатор заказа
- Идентификатор пользователя
- Список позиций (snapshot: товар, количество, цена на момент оформления)
- Общая сумма
- Текущий статус
- Идентификатор платежа (после успешной оплаты)
- Даты создания и обновления
<!-- Пока откажусь -->
- Адрес доставки
- История изменений статуса — список `(статус, время, причина?)`; нужна для gRPC-стрима и аудита

**Статусы:**

| Статус      | Описание                               |
| :---------- | :------------------------------------- |
| `PENDING`   | Заказ создан, ожидает обработки        |
| `RESERVED`  | Товары зарезервированы на складе       |
| `PAID`      | Оплата подтверждена (gRPC `Charge`)    |
| `CONFIRMED` | Склад списал товар (`ORDER_CONFIRMED`) |
| `CANCELLED` | Заказ отменён                          |

**Допустимые переходы:**

```
PENDING → RESERVED → PAID → CONFIRMED
         ↘          ↘
          CANCELLED  CANCELLED
PENDING → CANCELLED
RESERVED → CANCELLED
PAID → CANCELLED   (компенсация: Refund + Release)
```

Переходы вне этой схемы запрещены.

**Правила:**
- В заказе ≥ 1 позиция, сумма > 0.
- Адрес доставки обязателен при оформлении (`POST /orders`).
- Цены позиций фиксируются при создании заказа и не меняются.
- После `PAID` состав заказа не редактируется (допустима только отмена по правилам ниже).
- `payment_id` обязателен в статусе `PAID`.
- Каждое изменение статуса записывается в историю.

**Отмена заказа (DELETE `/orders/{id}`):**

| Текущий статус        | Действие                                                        |
| :-------------------- | :-------------------------------------------------------------- |
| `PENDING`, `RESERVED` | `RELEASE_RESERVATION` → `CANCELLED` (деньги не списаны)         |
| `PAID`                | Асинхронно: `RELEASE_RESERVATION` → gRPC `Refund` → `CANCELLED` |
| `CONFIRMED`           | Отмена запрещена (HTTP 400)                                     |

---

### 3. Позиция заказа (Order Item)

Товар в составе корзины или заказа.

**Атрибуты:**
- Идентификатор товара (ссылка на каталог Store)
- Количество
- Цена за единицу
- Общая стоимость позиции (количество × цена)

**Правила:**
- Количество ≥ 1, цена > 0.
- Общая стоимость = количество × цена за единицу.

## Варианты использования

### UC-1: Добавление товара в корзину

**Поток:**
1. Пользователь указывает товар и количество.
2. Order Service валидирует количество и цену.
3. Позиция добавляется в корзину (merge, если товар уже есть).
4. Order Service публикует `RESERVE_ITEMS`.
5. Store резервирует товар и публикует `ITEMS_RESERVED`.
6. Order Service обновляет состояние корзины.

**Ошибки:** недопустимое количество; резерв не удался — сообщение пользователю, позиция не добавляется (или удаляется из корзины).

---

### UC-2: Оформление заказа

**Предусловия:** корзина не пуста, резерв подтверждён, указан адрес доставки.

**Поток:**
1. Пользователь инициирует `POST /orders`.
2. Order Service создаёт заказ со статусом `PENDING`, фиксирует позиции и сумму.
3. Статус переходит в `RESERVED` (резерв уже получен на этапе корзины).
4. Корзина очищается.

**Ошибки:** пустая корзина, нет адреса, резерв не подтверждён.

---

### UC-3: Оплата заказа

**Предусловия:** заказ в статусе `RESERVED`.

**Поток:**
1. Пользователь инициирует оплату.
2. Order Service **синхронно** вызывает Payment Service (gRPC `Charge`).
3. При успехе: статус → `PAID`, сохраняется `payment_id`.
4. Order Service публикует `CONFIRM_ORDER`.
5. Store списывает товар и публикует `ORDER_CONFIRMED`.
6. Order Service переводит заказ в `CONFIRMED` и публикует `ORDER_FINALIZED`.

**Ошибки:** отказ платежа — заказ остаётся `RESERVED`, пользователь может повторить; таймаут gRPC — ошибка без смены статуса.

---

### UC-4: Компенсация при отказе склада

**Предусловия:** заказ в статусе `PAID`, отправлено `CONFIRM_ORDER`.

**Поток (сценарий B из DESIGN):**
1. Store не может списать товар и публикует `RESERVATION_FAILED`.
2. Order Service **синхронно** вызывает Payment Service (gRPC `Refund`).
3. Статус → `CANCELLED`.
4. Order Service публикует `ORDER_CANCELLED` (Notification, Analytics).

---

### UC-5: Отмена заказа пользователем

**Поток (сценарий C из DESIGN):**
1. Пользователь вызывает `DELETE /orders/{id}`.
2. Order Service проверяет текущий статус и выполняет действия из таблицы «Отмена заказа» (см. сущность Order).
3. Публикуется `ORDER_CANCELLED` при успешной отмене.

**Ошибки:** статус `CONFIRMED` — HTTP 400.

---

### UC-6: Отслеживание статуса (gRPC Streaming)

**Поток:**
1. Клиент подписывается на `WatchOrderStatus`.
2. Order Service отправляет текущий статус.
3. При каждом изменении статуса — новое сообщение в стриме.
4. Стрим закрывается при финальном статусе (`CONFIRMED` или `CANCELLED`) или отключении клиента.

**Ошибки:** заказ не найден.

## События NATS (исходящие из Order Service)

| Событие               | Когда                       |
| :-------------------- | :-------------------------- |
| `RESERVE_ITEMS`       | Добавление товара в корзину |
| `CONFIRM_ORDER`       | Успешная оплата             |
| `RELEASE_RESERVATION` | Отмена до или после оплаты  |
| `ORDER_FINALIZED`     | Заказ подтверждён складом   |
| `ORDER_CANCELLED`     | Отмена или компенсация      |

## Входящие события (Order Service слушает)

| Событие              | Действие                                             |
| :------------------- | :--------------------------------------------------- |
| `ITEMS_RESERVED`     | Обновить состояние корзины / резерва                 |
| `ORDER_CONFIRMED`    | Статус → `CONFIRMED`, опубликовать `ORDER_FINALIZED` |
| `RESERVATION_FAILED` | Компенсация (UC-4)                                   |

## gRPC (исходящие вызовы)

| Метод    | Когда                                       |
| :------- | :------------------------------------------ |
| `Charge` | UC-3, синхронно                             |
| `Refund` | UC-4, UC-5 (отмена после оплаты), синхронно |

## Вне scope Order Service

Следующее сознательно не описывается здесь — ответственность других сервисов или инфраструктуры:

- Каталог товаров, атрибуты (размер, цвет) — Store Service (MongoDB).
- Детали платёжных транзакций — Payment Service (PostgreSQL).
- Уведомления пользователю — Notification Service.
- Чеки и витрины — Analytics Service (MinIO, PostgreSQL).
- Доставка (`SHIPPED`), валюты, политики возврата после `CONFIRMED`.
- Retry, JetStream, circuit breaker — технические ADR, не бизнес-правила домена.
